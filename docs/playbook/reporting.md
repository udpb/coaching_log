# 메인 → 사용자 보고 포맷 (coaching-log)

> 사용자 강조 원칙: **투명한 보고**. 성공 위주 요약 금지.

---

## 5섹션 표준 포맷

```
## ✅ 한 일
- <구체적 — 파일:라인 · 명령 · 결과>

## ❌ 못한 일 / 보류
- <항목> — 사유

## 🤔 결정한 것
- <자율 판단 — 사용자가 뒤집을 수 있게>

## 🔬 검증
- <엔드포인트 호출 결과 · SQL 결과 · 브라우저 확인>
- <검증 못 한 것도 명시>

## ⚠️ 위험 신호 / 다음 진입점
- <다음 결정 · 기술 부채 · 다음 세션 진입점>
```

---

## 각 섹션 가이드

- **한 일**: "수정했어요" ❌ → "`api/extract-session.js:93` 에 JWT 검증 추가" ✅. 시간순.
- **못한 일**: 숨기지 않음. 사유 분류 (시간/스코프 · 의존성 · 결정 필요 · 모르는 것).
- **결정한 것**: 자율 판단 명시. ADR 후보면 표기.
- **검증**: 빌드가 없으므로 **무엇을 어떻게 확인했는지** 구체적으로. 예: "배포 후 `POST /api/extract-session` (no token) → 401 확인", "admin/pm/coach 3역할로 `coaching_logs` SELECT 매트릭스 확인". **검증 못 한 것 명시.**
- **위험 신호**: 다음 결정 · 기술 부채 · 다음 진입점.

## 중간 보고

```
[<단계 N/M>] <한 일 한 줄> · <검증 한 줄> · <다음 한 줄>
```

## 질문할 때
"어떻게 할까요?" ❌ → 옵션 A/B + 장단점 + 추천 + "진행할까요?".

## 피해야 할 패턴
1. 형용사("훌륭하게") 금지. 2. 에러 숨기기 금지. 3. "잘 됐을 거예요" → "검증 안 함" 명시. 4. TODO 무한 추가 금지. 5. 에이전트 결과 그대로 전달 금지 (메인 큐레이션).

> **신뢰는 투명성에서 나온다. 못 한 것을 못 했다고 말하라.**
