언더독스 · UD IMPACT — 개발 인수인계 개요

coaching·log

언더독스 코치가 창업팀 코칭 세션을 말(또는 녹음)로 기록하면, AI가 구조화된 일지로 정리하고 — 회차가 쌓이며 “이 팀이 나아지고 있나”를 팀별 타임라인으로 추적하는 도구.

주 사용자 · 코치 관리 · PM / admin 자매 제품 · coach-finder 빌드 없는 단일 index.html

왜 만들었나

코치가 “기록”이 아니라 “코칭”에 시간을 쓰게.

코칭이 끝나면 보통 무슨 얘기를 했는지 다시 타이핑하는 데 시간이 듭니다. coaching-log는 코치가 말한 내용을 붙여넣거나 녹음하면 AI가 알아서 핵심을 뽑아 구조화 일지로 정리합니다. 그래서 기록 노동은 줄고, 회차가 쌓일수록 한 팀의 이행률·에너지·반복되는 막힘이 한눈에 보여 “성장하고 있는가”를 데이터로 판단할 수 있습니다.

핵심 기능

5가지가 전부입니다.

🎙️

STT/음성 → AI 일지

붙여넣기 또는 최대 2시간 녹음(5분 청크 자동 전사). Gemini가 3-pass로 내러티브 + 22개 구조화 필드를 근거·신뢰도와 함께 추출.

📈

성과 추적 대시보드

팀별 타임라인 · 평균 이행률 · 에너지 추이 · 반복 막힘 경고 · 지표 시계열 차트 · 근본이슈 변화.

🎯

프로젝트별 KPI

PM이 프로젝트마다 필수 KPI 지정 → 일지 폼에 자동 prefill. 추출도 한국어 지표명으로 매칭(중복 제거).

🙋

코치 본인 정보 관리

프로필·계약정보(주소·계좌·사업자) 자가편집 + 신규 코치 강제 온보딩.

🗂️

사업·배정·평가

admin/pm이 사업기획(business_plans)·코치 배정·5축 평가. 코치 본인은 자기 평가 조회 불가(솔직성).

📄

리포트 생성

세션별/번들 리포트(HTML)로 내보내기 — 근거·원문 부록 옵션.

기술 스택

단순함이 곧 설계입니다 — 빌드 도구가 없습니다.

프론트 바닐라 JS · 단일 index.html (~10,880줄) 필드 SoT field-defs.js 서버 Vercel serverless 1개 api/extract-session.js DB Supabase (Postgres · RLS · pgvector) AI Gemini 2.5-pro → flash 폴백 인증 브라우저 anon 키 + RLS

왜 빌드가 없나

번들러·프레임워크 없이 index.html 하나 + CDN 스크립트로 동작합니다. 받자마자 브라우저로 열어 동작을 볼 수 있고, 배포는 정적 파일 + 함수 1개뿐이라 운영이 단순합니다. (React 도입은 ADR 필요)

지켜야 할 4가지

① 사용자 입력은 항상 escHtml/escAttr로 이스케이프(XSS). ② RLS가 진짜 보안 경계 — UI 권한 체크는 보조. ③ DB 변경은 새 마이그레이션 파일만(기존 수정 금지). ④ 22필드 규칙은 field-defs.js에서 생성.

개발 방식

“기능을 만드는 자”와 “구조를 지키는 자”를 나눕니다.

🧭 메인 세션 (구조를 지킴)

설계(Architect) · 문서 정합·보호(Guardian) · 자료 정리(Curator) · 위임·검증(Orchestrator) · 기록(Historian). 직접 구현은 하지 않음.

🛠️ 서브 에이전트 (기능을 만듦)

자급자족 브리프(.claude/agent-briefs)를 받아 구현 → 5섹션으로 보고. 메인이 git diff로 직접 검증.

📌 결정은 ADR

docs/decisions — “왜 이렇게 했나”. Accepted 후 수정 금지(Supersede).

📖 용어는 glossary

docs/glossary.md — 코드·문서·라벨이 같은 용어를 쓰도록 하는 단일 진실.

🧪 검증은 직접

빌드/테스트가 적어 — 엔드포인트 실호출 · RLS 매트릭스 · 브라우저 육안으로 증거를 만듦.

두 제품의 관계

coach-finder와 한 몸 — 하나의 Supabase를 공유합니다.

coach-finder

PM이 코치를 찾고 사업을 기획·수주·계약. 사업의 라이프사이클을 관리(진실원천).

coaching-log (이 제품)

코치가 세션을 기록·추적. 사업 상태를 추종 — 사업이 “진행중(active)”이 되면 트리거가 여기 프로젝트를 자동 생성.

🔗 공유: 하나의 Supabase + coaches_directory 마스터 테이블 · 같은 이메일 = 같은 사람 = 같은 역할(admin/pm/coach)

• 스키마(마이그레이션) 진실원본 = coaching-log (공유 테이블 변경은 여기서). • 사업 status 진실원본 = coach-finder (ADR-023 — planning→active→completed/cancelled, coaching-log는 추종). • coaches_directory의 embedding·추론스킬·역할태그 컬럼은 coach-finder 소유 기능(여기선 미사용, ADR-025).
