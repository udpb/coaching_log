# ADR-021: 2시간 코칭 세션 지원 — 클라이언트 청크 분할 전사 아키텍처

## Context (왜)
현재 녹음→추출은 오디오를 base64 로 요청 본문에 실어 보낸다 (인라인 한도 4M chars ≈ 3MB ≈ Opus 6~7분). 코칭은 최대 2시간 — 2시간 오디오는 32kbps 기준 ~29MB 로:
1. **Vercel 요청 본문 4.5MB 하드 리밋** 초과 (플랜 무관, 변경 불가)
2. Gemini inline_data 20MB 한도 초과 (Files API 필요)
3. 단일 호출 전사 출력(~2.5만+ 토큰)이 현 maxOutputTokens 초과
반면 **텍스트 전사 경로는 이미 2시간을 커버** — 한국어 2h 발화 ≈ 25k~50k 자 < 120k 자 한도.

## Decision (무엇을)
**클라이언트가 녹음을 5분 청크로 분할해 청크별로 전사(transcribe-only, flash)하고, 전사 텍스트를 누적한 뒤 기존 텍스트 추출 경로로 마무리한다.**
- MediaRecorder 를 5분마다 stop/start (timeslice 청크는 헤더가 첫 청크에만 있어 단독 재생 불가 — 재시작 방식으로 자립형 webm 확보). 32kbps mono → 청크당 ~1.2MB (base64 ~1.6MB, 한도의 40%).
- `/api/extract-session` 에 `task: 'transcribe'` 추가 — 오디오만 받아 verbatim 전사 JSON 반환. 모델 = **gemini-2.5-flash 고정** (전사에 pro 불필요 — 비용·쿼터 보호), thinking 0, 비스트림.
- 청크 전사 결과를 STT textarea 에 실시간 누적 (초안 자동저장이 텍스트를 보호). 녹음 종료 → 마지막 청크 전사 → 기존 text 모드 추출(pro) 자동 실행.
- 2시간 자동 종료 + 110분 경고. 청크 실패 시 자동 재시도 2회 + 수동 재시도.
- `vercel.json` maxDuration 60→300 (긴 전사·추출 여유; Fluid compute 기준 Hobby 도 300 허용).

## Consequences (결과)
### Positive
- 플랫폼 한도 안에서 세션 길이 사실상 무제한 (텍스트 120k 자 = 4~5시간 분량).
- 라이브 증분 전사 — 세션 끝나면 전사가 이미 끝나 있어 대기 시간 최소화.
- 비용/쿼터 효율: 전사는 flash(저가·고쿼터), 추출만 pro 1회.
- 중간 유실 내성: 전사된 텍스트는 draft 자동저장에 즉시 보호.
### Negative / Trade-off
- 청크 경계(stop/start ~수십 ms)에서 단어 잘릴 수 있음 — 전사 품질에 미미.
- 화자 구분이 청크별로 독립 수행 — 청크 간 화자 일관성은 보장 안 됨.
- 클라이언트 상태 머신 복잡도 증가 (업로드 큐·재시도).
### 동시성 (사용자 질문 — 현재 기준 최대 동시 인원)
세션당 부하: 전사 24회(flash, 5분당 1회 ≈ 0.2 RPM/인) + 추출 1회(pro).
- **Gemini 무료 티어** (flash 10 RPM·250 RPD): 청크 타이밍이 겹치는 최악 분에 N 요청 → **동시 ~10명**, 일일 총량은 RPD 가 병목 → **~10세션/일**.
- **유료 Tier 1** (flash 1,000 RPM·10k RPD): **동시 100명+ 무리 없음**, ~400세션/일.
- Vercel 동시성·함수시간은 병목 아님 (청크당 10~30초).
→ 실사용 확장 시 Gemini 키 유료 티어 전환이 선결 조건.

## Alternatives Considered
| 옵션 | 설명 | 왜 채택 안 됐나 |
|------|------|-----------------|
| Gemini Files API + Supabase Storage 경유 | 2h 파일 업로드 후 file_uri 참조 | 서버가 30~60MB 중계(다운로드+업로드) — 함수 시간·메모리 위험, 스토리지 비용·수명주기 관리 추가. 청크 방식이 더 단순·견고 |
| 브라우저에서 Gemini Files API 직접 업로드 | 중계 제거 | **API 키 클라이언트 노출 — 절대 금지** |
| 본문 한도 상향 | 4.5MB는 Vercel 하드 리밋 | 불가능 |
| 클라이언트 Web Speech API 실시간 STT | 서버 비용 0 | 품질·브라우저 편차, 화자 구분 불가, 기존 Gemini 전사 품질 포기 |

## Addendum (2026-06-12 — append only, 원문 불변)
실사용 첫 테스트에서 Vercel 플랫폼 413 `FUNCTION_PAYLOAD_TOO_LARGE` 발생. 원인: `audioBitsPerSecond: 32000` 은 **힌트일 뿐** — 브라우저가 무시하면(기본 96~128kbps, Safari AAC 256kbps) 5분 청크가 4.5MB 본문 한도를 초과한다. 보완: **회전 트리거를 "시간(5분) ∥ 용량(2MB) 중 선도달" 로 이중화** (15초 간격 `requestData()` 계측 — 청크 경계는 여전히 recorder 인스턴스 교체라 자립형 webm 원칙 불변). 업로드 직전 base64 3.9M 안전판 + 413 사용자 메시지 매핑 추가. 다국어(한/영 혼용) 전사 품질은 동일 테스트에서 정상 확인됨.
