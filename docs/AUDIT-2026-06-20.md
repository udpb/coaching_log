## ⓐ 죽은 컬럼/테이블 — "죽음"이 아니라 "미구현" 또는 "타 제품 소유"

본 레포는 **두 제품(coaching-log + coach-finder)의 스키마 SoT**다. 따라서 coaching-log 코드가
안 읽는 컬럼이라도 (1) coach-finder 가 쓰거나 (2) 아직 안 만든 기능의 자리일 수 있다.
무조건 삭제는 **공유 계약 위반 + RLS/외부 도구 파괴** 위험. 분류:

### A. 미구현 기능 스캐폴딩 (coaching-log·coach-finder 양쪽 다 코드 없음 — 진짜 정리 후보)
| 컬럼 | 테이블 | 추가 마이그 | 의도(주석) | 현황 |
|------|--------|-----------|-----------|------|
| `embedding` (vector 1536) | coaches_directory | 20260424_phase4e | RAG 코치 추천(PM향) | RPC `search_coaches_by_embedding()` 정의만, 호출 0 |
| `embedding_source_hash` | coaches_directory | phase4e | 임베딩 무효화 추적 | 미사용 |
| `embedding_updated_at` | coaches_directory | phase4e | 신선도 | 미사용 |
| `embedding_model` | coaches_directory | phase4e | 모델 감사기록 | 미사용 (⚠️ 주석 "OpenAI" 오기) |
| `inferred_skills` (text[]) | coaches_directory | 20260504_phase_e4 | 로그→자동 스킬추론 | 외부 도구 미연동, 호출 0 |
| `inferred_skills_updated_at` | coaches_directory | phase_e4 | 신선도 | 미사용 |
| `inferred_skills_source` | coaches_directory | phase_e4 | gemini\|frequency | 미사용 |
| `roles_capable` (text[]) | coaches_directory | 20260523_phase_p | 보유 역량 역할 | GIN 인덱스만, coaching-log 호출 0 |
| `roles_active_2026` (text[]) | coaches_directory | phase_p | 2026 활동 의향 | coaching-log 호출 0 |
| `ud_programs` (text[]) | coaches_directory | 20260523_phase_q | 참여 UD 프로그램 | coaching-log 호출 0 |

→ **embedding(4) + inferred_skills(3) = 7개**는 본 레포가 만든 미완 기능. **roles/ud_programs(3)**는
coach-finder 필터용으로 추정 — coach-finder 코드 확인 전엔 단정 불가.

### B. coach-finder 도메인 (본 레포 코드는 안 쓰지만 coach-finder 가 씀 — 삭제 금지)
- `business_plan_coaches`: `payment_info`(jsonb) · `task_summary`(text)
- `business_plans`: `client` · `total_budget` · `kickoff_count` · `task_program_learning` ·
  `task_review_participation` · `task_program_attendance` · `task_participant_type` ·
  `task_report_frequency` · `payment_schedule` · `installment_midpay_note` · `installment_balance_note`
  (계약서 생성·결제 옵션 — coach-finder/계약생성기 소유)
- `business_plans.legacy_firestore_id`: Firestore→Supabase 1회성 이관 키(외부 스크립트)
- `api_consumers` · `api_consumer_usage` 테이블: API 키 관리(admin 전용, 별도 패널)

→ 이들은 **공유 계약**. 삭제하려면 coach-finder 동기화 + ADR + 마이그레이션 필요. **현 상태 유지가 정답.**

### C. 생존 확인 (오탐 방지 — 본 레포가 실제 사용)
`required_kpis`(ADR-022) · `extraction_model`/`extraction_version`(phase_aa) ·
`is_business`/`business_number`/`business_name`(계약정보 모달) · `business_plan_id`(트리거 링크) ·
`onboarded_at`(온보딩 게이트) · `coach_id`(phase_z SoT). — 전부 생존.

### 🔶 ⓐ 사용자 결정 필요
embedding/RAG·inferred_skills 기능을 **(1) 계속 계획**(컬럼 보존·기능 향후 구현) vs
**(2) 폐기**(ADR + drop 마이그레이션으로 정리)? roles/ud_programs 는 coach-finder 확인 후 판단.
**기본 권장:** 보존 + 의도 주석(이미 있음). 미사용 컬럼은 800행 규모에 비용 무시 가능 · 삭제 리스크>이득.

## ⓒ 스키마 SoT 드리프트 — ✅ 정합 (모순 0)

42개 마이그레이션 시간순 정독 결과 **모순/누락 0**. 핵심:
- status 라이프사이클이 phase5b(5값)→phase_f(8값 혼재)→phase_ac(won+active 과도기)→
  phase_ad(데이터 변환)→**phase_ae(4값 정본 planning/active/completed/cancelled)** 로 수렴. 최종 승자 명확.
- 트리거 최종형: `bp_lifecycle_sync_ins`(AFTER INSERT, active+project_id NULL) ·
  `bp_lifecycle_sync_upd`(AFTER UPDATE, status 변동+active+NULL) ·
  `bp_status_propagate_upd`(completed→closed/cancelled→archived). 함수 `handle_business_plan_won()` ·
  `handle_business_plan_terminal()`. 모두 멱등(DROP IF/IF NOT EXISTS/CREATE OR REPLACE).
- RLS 헬퍼 5종 + `can_access_coach_contract_info()` 전부 안정.

### 제로베이스 재생 한계 (문서화됨 · 의도적)
`20260610_phase_z` 헤더가 명시: phase4a(0421)가 `coach_id = auth.uid()` 정책을 만드는데 coach_id 는
phase_z(0610)에서야 추가됨 → 빈 DB 시간순 재생 시 phase4a 실패. "적용 마이그 수정 금지" 원칙상
phase4a 못 고침. **라이브는 IF NOT EXISTS 로 무영향**, 제로베이스 재구축만 별도 절차 필요. = 버그 아님.

### 🔷 라이브 검증 SQL (사용자 실행 — 메인은 Supabase 접근 불가)
```sql
-- 1) status 4값만 (won/draft/proposed/lost = 0 이어야)
SELECT status, count(*) FROM public.business_plans GROUP BY status ORDER BY status;
-- 2) active 인데 project 없는 행 0
SELECT count(*) FROM public.business_plans WHERE status='active' AND project_id IS NULL;
-- 3) business_plan_id NULL 인 projects (HANDOFF "bp없는 세션有 projects" 확인)
SELECT id, name, status FROM public.projects WHERE business_plan_id IS NULL;
-- 4) status CHECK 가 4값인지
SELECT pg_get_constraintdef(oid) FROM pg_constraint WHERE conname='business_plans_status_check';
-- 5) 트리거 3종 존재
SELECT tgname FROM pg_trigger WHERE tgrelid='public.business_plans'::regclass AND NOT tgisinternal;
```
